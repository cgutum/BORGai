## Executive Summary

This document describes an **XGBoost-based time series forecasting system** designed to predict weekly core supply quantities for automotive turbocharger remanufacturing operations. The model:

- **Predicts**: Weekly supply quantities by core type and vehicle model
- **Horizon**: 26 weeks (6 months) into 2026
- **Granularity**: Weekly forecasts with 80% and 95% confidence intervals
- **Training**: 2020-2024 data (471 records)
- **Validation**: 2025 data (144 records)
- **Features**: 32 engineered features including temporal, lagged supply, EWM, registration lags, orders, and reusability metrics

**Key Results:**
- Training: 2020-2024 (all historical data)
- Validation MAE: 2.12 units, RMSE: 4.05 units, MAPE: 30.79% (2025 forecast vs actual)
- Feature importance: EWM features dominate (24.0% + 13.2% = 37.2%)
- Seamless timeline: 2025 validation → 2026 forecast
- Confidence intervals: ±2σ for 95%, ±1.28σ for 80%

## Problem Definition

### Business Context

In automotive remanufacturing, **core supply** (returned defective parts) is the critical input to the remanufacturing process. Accurate forecasting enables:

1. **Capacity Planning**: Optimize facility and labor allocation
2. **Inventory Management**: Balance core storage costs vs. remanufacturing demand
3. **Procurement Strategy**: Plan for supplementary core purchases
4. **Production Scheduling**: Align remanufacturing output with demand

### Technical Challenge

Core supply forecasting is complex due to:

- **Long lag times**: 5-6 years between vehicle sale and turbocharger failure
- **Multi-channel returns**: Workshop, fleet, and private channels with different lag distributions
- **Seasonality**: Monthly and quarterly patterns in vehicle registrations and failures
- **Multiple products**: 3 core types (BMW X3 2018, BMW X3 2020, BMW X5 2021) with distinct lifecycles

### Forecasting Objective

**Primary Goal:** Predict weekly core supply quantities for the next 6 months (26 weeks) in 2026, broken down by:
- **Core Type** (turbocharger model: TBLBMWX32018, TBLBMWX32020, TBLBMWX52021)
- **Vehicle Model** (BMW X3 2018, BMW X3 2020, BMW X5 2021)

**Output:** Point forecasts with 80% and 95% prediction intervals

**Timeline:**
- **Training**: 2020-2024 (build model on all historical data)
- **Validation**: 2025 (forecast vs actual to test model performance)
- **Forecast**: 2026 (26-week seamless prediction with confidence bands)

## Feature Engineering

The model uses **29 features** across 7 categories:

### 1. Temporal Features (8 features)

Capture time-based patterns and seasonality.

| Feature | Description | Type |
|---------|-------------|------|
| `Year` | Calendar year | Integer |
| `Month` | Calendar month (1-12) | Integer |
| `Week` | ISO week number (1-52) | Integer |
| `Quarter` | Calendar quarter (1-4) | Integer |
| `Month_Sin` | sin(2π × Month / 12) | Float |
| `Month_Cos` | cos(2π × Month / 12) | Float |
| `Week_Sin` | sin(2π × Week / 52) | Float |
| `Week_Cos` | cos(2π × Week / 52) | Float |

**Rationale:** 
- Linear time features capture trends
- Cyclical encoding (sin/cos) captures seasonality without artificial discontinuities (December→January)

### 2. Lagged Supply Features (9 features)

Capture recent supply history and momentum.

| Feature | Description | Lookback |
|---------|-------------|----------|
| `Target_Lag_1W` | Supply 1 week ago | 1 week |
| `Target_Lag_2W` | Supply 2 weeks ago | 2 weeks |
| `Target_Lag_3W` | Supply 3 weeks ago | 3 weeks |
| `Target_Lag_4W` | Supply 4 weeks ago | 4 weeks |
| `Target_Lag_8W` | Supply 8 weeks ago | 8 weeks |
| `Target_Lag_12W` | Supply 12 weeks ago | 12 weeks |
| `Target_RollingMean_4W` | 4-week rolling average | 4 weeks |
| `Target_RollingStd_4W` | 4-week rolling std dev | 4 weeks |
| `Target_RollingMean_12W` | 12-week rolling average | 12 weeks |
| `Target_EWM_4W` | 4-week exponential weighted moving avg | 4 weeks |
| `Target_EWM_12W` | 12-week exponential weighted moving avg | 12 weeks |

**Rationale:** 
- Short lags (1-4 weeks) capture immediate momentum
- Medium lags (8-12 weeks) capture medium-term trends
- Rolling statistics smooth noise and capture volatility

### 3. Registration Lag Features (2 features)

Capture vehicle sales that will result in failures.

| Feature | Description | Lookback |
|---------|-------------|----------|
| `Lagged_Reg_5Y` | Registrations 5 years ago | 60 months |
| `Lagged_Reg_6Y` | Registrations 6 years ago | 72 months |

**Rationale:** 
- Turbochargers typically fail after 5-6 years (peak of Weibull distribution, β=2.5, η=5.0, γ=1.0)
- 5-year lag aligns with peak failure rate
- 6-year lag captures tail of failure distribution

**Mathematical Foundation:**

The Weibull failure distribution is:

$$
f(t) = \frac{\beta}{\eta} \left( \frac{t - \gamma}{\eta} \right)^{\beta - 1} \exp\left[ -\left( \frac{t - \gamma}{\eta} \right)^{\beta} \right]
$$

With β=2.5, η=5.0, γ=1.0, the peak occurs at:

$$
t_{\text{mode}} = \gamma + \eta \left( \frac{\beta - 1}{\beta} \right)^{1/\beta} \approx 5 \text{ years}
$$

### 4. Order Features (7 features)

Capture recent order patterns as leading indicators.

| Feature | Description | Lookback |
|---------|-------------|----------|
| `Order_Qty` | Current week orders | 0 weeks |
| `Order_Qty_Lag_4W` | Orders 4 weeks ago | 4 weeks |
| `Order_Qty_Lag_8W` | Orders 8 weeks ago | 8 weeks |
| `Order_Qty_Lag_12W` | Orders 12 weeks ago | 12 weeks |
| `Order_RollingMean_8W` | 8-week rolling average | 8 weeks |
| `Order_RollingMean_12W` | 12-week rolling average | 12 weeks |
| `Order_RollingStd_8W` | 8-week rolling std dev | 8 weeks |

**Rationale:** 
- Orders (failures) precede supply returns by 4-12 weeks (log-normal lag distribution)
- Lagged orders correlate with future supply

### 5. Reusability Feature (1 feature)

| Feature | Description |
|---------|-------------|
| `Avg_Reusability_Rate` | Average component reusability % |

**Rationale:** 
- Higher reusability → more valuable cores → potentially different return behavior
- Captures intrinsic core quality differences

### 6. Trend Feature (1 feature)

| Feature | Description |
|---------|-------------|
| `Weeks_Since_Start` | Weeks elapsed since dataset start |

**Rationale:** 
- Captures long-term trends not explained by seasonal patterns

### 7. Categorical Features (2 features)

| Feature | Description | Encoding |
|---------|-------------|----------|
| `Core_Type_Encoded` | Turbocharger model | Label encoding |
| `Vehicle_Model_Encoded` | Vehicle model | Label encoding |

## Model Architecture

### XGBoost Configuration

```python
model = xgb.XGBRegressor(
    n_estimators=500,        # Number of boosting rounds (increased for better learning)
    max_depth=4,             # Maximum tree depth (reduced to prevent overfitting)
    learning_rate=0.03,      # Step size shrinkage (η) - lower for finer adjustments
    subsample=0.7,           # Row sampling ratio (more aggressive)
    colsample_bytree=0.7,    # Column sampling ratio
    colsample_bylevel=0.7,   # Column sampling per level
    min_child_weight=5,      # Minimum sum of instance weight (increased)
    gamma=0.2,               # Minimum loss reduction for split (more conservative)
    reg_alpha=0.3,           # L1 regularization (Lasso) - stronger
    reg_lambda=2.0,          # L2 regularization (Ridge) - stronger
    random_state=42,
    n_jobs=-1,
    early_stopping_rounds=50 # Stop if no improvement
)
```

### Hyperparameter Rationale

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `n_estimators` | 150 | Sufficient boosting rounds with early stopping |
| `max_depth` | 5 | Balances complexity vs. overfitting; captures feature interactions |
| `learning_rate` | 0.05 | Lower rate (vs. 0.1) with more trees → better generalization |
| `subsample` | 0.8 | Row sampling reduces overfitting, adds stochasticity |
| `colsample_bytree` | 0.8 | Column sampling prevents over-reliance on single features |
| `min_child_weight` | 3 | Prevents splits on sparse data |
| `gamma` | 0.1 | Regularizes by requiring minimum loss reduction |
| `reg_alpha` | 0.1 | L1 regularization encourages feature sparsity |
| `reg_lambda` | 1.0 | L2 regularization smooths predictions |

### Why XGBoost?

1. **Handles Mixed Feature Types**: Seamlessly combines continuous (lags, registrations) and categorical (core type) features
2. **Nonlinear Relationships**: Captures complex interactions (e.g., seasonality × core type)
3. **Robustness**: Resistant to outliers and missing values
4. **Interpretability**: Feature importance and SHAP values
5. **Efficiency**: Fast training and prediction with parallel processing
6. **Regularization**: Built-in L1/L2 regularization prevents overfitting

### Early Stopping

```python
model.fit(
    X_train, y_train,
    eval_set=[(X_train, y_train), (X_val, y_val)],
    verbose=False
)
```

Training stops if validation error doesn't improve for 50 consecutive rounds (early_stopping_rounds=50), preventing overfitting. Model typically converges around iteration 250-300.

## Forecasting Methodology

### Iterative Multi-Step Forecasting

For each forecast week in 2027:

1. **Construct Features**: Using historical data up to forecast date
2. **Lagged Features**: Use actual historical values or prior forecasts (for lags)
3. **Registration Lags**: Lookup actual registrations from 5-6 years prior
4. **Predict**: Generate forecast for that week
5. **Store**: Add forecast to history for subsequent forecasts

### Feature Construction for Forecasting

# Lagged supply (from historical actuals)

1. **Missing Lags**: For first few forecast weeks, some lags extend beyond available history → use 0 or averages
2. **Order Proxies**: Future orders unknown → use recent historical averages (conservative assumption)

## Prediction Intervals

### Methodology

Prediction intervals quantify forecast uncertainty using **residual-based estimation**.

#### Step 1: Calculate Validation Residuals

```python
residuals = df_val['Target'] - df_val['Predicted']
residual_std = np.std(residuals)
```

#### Step 2: Compute Confidence Intervals

Assuming residuals are normally distributed (Gaussian assumption):

- **95% CI**: Forecast ± 2σ  
  ```python
  Lower_95 = Forecast - 2 * residual_std
  Upper_95 = Forecast + 2 * residual_std
  ```

- **80% CI**: Forecast ± 1.28σ  
  ```python
  Lower_80 = Forecast - 1.28 * residual_std
  Upper_80 = Forecast + 1.28 * residual_std
  ```

**Mathematical Foundation:**

For a normal distribution N(μ, σ):
- 95% of values fall within [μ - 1.96σ, μ + 1.96σ] ≈ [μ - 2σ, μ + 2σ]
- 80% of values fall within [μ - 1.28σ, μ + 1.28σ]

#### Step 3: Apply Non-Negativity Constraint

```python
df_forecast['Lower_95'] = (df_forecast['Forecast'] - 2 * residual_std).clip(lower=0)
df_forecast['Upper_95'] = (df_forecast['Forecast'] + 2 * residual_std)
```

Supply quantities cannot be negative, so lower bounds are clipped at zero.

### Interpretation

- **80% CI**: "We are 80% confident actual supply will fall within this range"
- **95% CI**: "We are 95% confident actual supply will fall within this range"

**Example:**
- Forecast: 50 units
- Residual σ: 10 units
- 95% CI: [30, 70] units
- 80% CI: [37, 63] units

### Limitations

1. **Gaussian Assumption**: Residuals may not be perfectly normal (can be validated with Q-Q plots)
2. **Constant Variance**: Assumes homoscedasticity (variance doesn't change over time)
3. **Point Estimate**: Intervals derived from point forecast, not full predictive distribution

**Future Enhancements:**
- Quantile regression for asymmetric intervals
- Conformal prediction for distribution-free intervals
- Bootstrapping for empirical intervals

## Performance Metrics

### Evaluation Metrics

#### 1. Mean Absolute Error (MAE)

$$
\text{MAE} = \frac{1}{n} \sum_{i=1}^{n} |y_i - \hat{y}_i|
$$

**Interpretation:** Average absolute difference between predicted and actual supply (in units).

**Example:** MAE = 5.2 units → On average, predictions differ from actuals by 5.2 units.

#### 2. Root Mean Squared Error (RMSE)

$$
\text{RMSE} = \sqrt{\frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2}
$$

**Interpretation:** Square root of average squared errors (in units). Penalizes large errors more than MAE.

**Example:** RMSE = 7.8 units → Typical prediction error is 7.8 units, with larger errors weighted more.

#### 3. Mean Absolute Percentage Error (MAPE)

$$
\text{MAPE} = \frac{100}{n} \sum_{i=1}^{n} \left| \frac{y_i - \hat{y}_i}{y_i} \right|
$$

**Interpretation:** Average percentage error. Scale-independent metric.

**Example:** MAPE = 18.5% → On average, predictions differ from actuals by 18.5%.

**Caveat:** MAPE is undefined when $y_i = 0$ and biased toward underforecasting.

**Implementation Note:** This project uses a **safe MAPE** calculation that excludes values below a threshold (1.0 unit) to prevent division-by-near-zero errors. Weekly supply can be 0-3 units for low-volume periods, which would cause standard MAPE to explode. The safe implementation:

```python
def safe_mape(y_true, y_pred, threshold=1.0):
    mask = np.abs(y_true) >= threshold
    if mask.sum() == 0:
        return 0.0
    return np.mean(np.abs((y_true[mask] - y_pred[mask]) / y_true[mask])) * 100
```

This approach filters out low-magnitude predictions while preserving meaningful percentage error metrics for substantial supply weeks.

#### 4. R² (Coefficient of Determination)

$$
R^2 = 1 - \frac{\sum_{i=1}^{n} (y_i - \hat{y}_i)^2}{\sum_{i=1}^{n} (y_i - \bar{y})^2}
$$

**Interpretation:** Proportion of variance explained by the model (0 to 1, higher is better).

**Example:** R² = 0.78 → Model explains 78% of variance in supply quantities.

## Appendix C: Key Assumptions

1. **Historical Patterns Persist**: Future supply patterns resemble historical patterns (stationarity assumption)
2. **Weibull Failure Model**: Turbocharger failures follow Weibull distribution (β=2.5, η=5.0, γ=1.0)
3. **Log-Normal Return Lags**: Return times follow log-normal distributions (Werkstatt 15d, Flotte 45d, Privat 90d)
4. **No External Shocks**: No major disruptions (pandemics, supply chain collapse, policy changes)
5. **Return Rate Stability**: 65-75% return rate remains constant
6. **Feature Availability**: All input features available at forecast time (registrations, component data)
